# quidproquo：把做過的事，換成別人也能用的東西

如果你也有寫筆記的習慣，應該很熟悉這種狀況：某天解掉一個問題，當下覺得「這個一定要記下來」，於是它被放進 Notion、GitHub issue、本機 markdown、Slack 訊息，或某個只會被自己遺忘的資料夾。

幾個月後，問題又出現一次。你知道自己曾經解過，但找不到。更糟的是，你只記得「我好像懂過」。

`quidproquo.cc` 是我為了解決這件事做出來的部落格。

它不是一個很精緻的個人品牌頁，也不是想要包裝成某種權威媒體。比較準確地說，它是一個低摩擦的公開筆記系統：把我在 AI、工程、產品、學習和生活裡真正踩過、想過、做過的事情，整理成一週後的自己還看得懂、其他人也可能用得上的形式。

## 為什麼叫 quidproquo

Quid pro quo 是拉丁語，意思是「以某物換某物」，也就是等價交換。

我喜歡這個名字，不是因為它聽起來比較酷，而是因為它剛好描述了我對寫作的期待。

很多經驗本來只屬於個人：一個 debug 過程、一個產品決策、一段自學路徑、一次攀岩時身體記住的失敗。如果它們只停在腦中，價值很快就會消散。寫下來，是把這些經驗換成可以被重新使用的東西。

但我更喜歡另一種版本：不是拿到十就還十，而是如果我從某件事得到十，就加上一點自己的理解，變成十一還出去。

所以這個部落格的核心不是「保存」，而是「多給一點」。

## 這裡會寫什麼

目前 `quidproquo.cc` 最多的內容集中在 AI 和工程，特別是 RAG、agent、context engineering、LLM observability、workflow orchestration 這些我實際工作和研究中反覆遇到的主題。

但它不只是一個技術部落格。

我也會寫產品設計、教育、自主學習、side project、攀岩、衝浪、咖啡。這些看起來分散，但對我來說其實是同一件事：我在意人怎麼學習、怎麼行動、怎麼把抽象想法做成能真的發生的東西。

技術文章會盡量具體，因為抽象的「心得」很難重用。能放錯誤訊息就放錯誤訊息，能寫命令就寫命令，能說清楚 trade-off 就不要只說「看情況」。我希望這些文章不是讀完覺得好像有道理，而是在你遇到同一個坑時，可以少花一點時間。

非技術文章則比較像保留生活裡那些不應該被工作吞掉的部分。攀岩、衝浪、咖啡都很誠實，它們會直接反映一個人怎麼面對限制、恐懼、節奏和失敗。這些東西不一定能變成方法論，但值得被記錄。

## 為什麼不用 Medium、Notion 或 Substack 就好

我試過。

問題不是它們不好，而是它們對我來說都有一點摩擦。要切換工具、登入、適應編輯器、處理格式，這些小事單獨看都不嚴重，但加起來足以讓「我應該把這個寫下來」消失。

我需要的是一個更貼近工作流的系統。

所以 `quidproquo.cc` 是用 Astro 和 Cloudflare Workers 做的。文章本體是 Markdown，放在 repo 裡；部落格是網站，也是作品集；內容可以被版本控制，也可以接到後續的搜尋、RAG、資料庫同步和自動化流程。

這聽起來比直接開 Medium 麻煩，但對我來說反而更低摩擦。因為我每天本來就在 editor、terminal、git 裡工作。寫文章不需要離開原本的環境。

## 真正重要的不是技術棧，是降低寫作阻力

這個部落格背後有一個我很在意的設計：post skill。

簡單說，我可以在解完一個問題後，直接對 agent 說「寫成文章」。它會根據內容判斷分類、套模板、補 frontmatter、存成 Markdown，接著跑檢查。這不是為了炫技，而是為了解決一個很現實的問題：

寫作最難的常常不是寫，而是開始。

如果每次都要重新決定標題、分類、格式、檔名、metadata，很多文章根本不會出生。把這些固定成本交給工具，人才比較有餘裕去處理真正重要的部分：這件事到底學到了什麼？哪裡值得別人知道？我能不能說得更清楚？

對我來說，AI agent 最有價值的地方之一，不是替人產生更多內容，而是幫人把已經發生過的經驗整理成可累積的資產。

## 這個部落格想成為什麼

我希望 `quidproquo.cc` 變成一個能長期累積的地方。

它可以是 debug archive，記錄那些我不想再查第二次的錯誤。

它可以是 AI engineering notebook，把 RAG、agent、evaluation、context engineering 這些概念整理到足夠實作的程度。

它可以是 product journal，記錄我怎麼看待學習平台、社群產品、行為設計和 side project。

它也可以只是生活紀錄，提醒我自己不是只由工作和技術組成。

更重要的是，它應該能被別人使用。你可能是剛開始做 RAG 的工程師、正在研究 agent workflow 的產品人、想做 side project 但不知道怎麼收斂的人，或只是剛好踩到同一個 bug。只要其中一篇文章讓你少繞一點路，這個部落格就有意義。

## 最後

我越來越覺得，公開寫作不是把自己包裝得更完整，而是把不完整的思考放到可以被修正、被引用、被交換的位置。

`quidproquo.cc` 對我來說就是這樣的地方。

把做過的事寫下來，換成別人也能用的東西。然後如果可以，多給一點。
