## ADDED Requirements

### Requirement: regenerateFromStep entry point

The kernel SHALL expose `regenerateFromStep({ flow_run_id, step_id, policy_override?, provider_override? }): Promise<RegenerationResult>` as the single public entry point for partial artifact regeneration. The call SHALL look up every `artifact_versions` row whose `flow_run_id` equals the supplied value, replay the named step via the `agent-flow` runtime with the optional `policy_override` / `provider_override` applied, patch the affected sections per the rules in the requirements below, and persist a new `artifact_versions` row chained to the prior version via `parent_version_id`. The result SHALL contain `{ new_version_id, affected_section_ids: string[], skipped_section_ids: string[] }`.

#### Scenario: Regenerate produces new version row

- **WHEN** `regenerateFromStep({ flow_run_id: 'R', step_id: 'search' })` is invoked and an artifact version `V_old` exists for run `R` with at least one section whose `flow_step_id='search'`
- **THEN** the kernel SHALL insert a new `artifact_versions` row `V_new` with `parent_version_id=V_old.id`, `flow_run_id='R'`, and the result SHALL return `{ new_version_id: V_new.id, ... }`

#### Scenario: Regenerate on unknown step rejected

- **WHEN** `regenerateFromStep({ flow_run_id: 'R', step_id: 'nonexistent' })` is invoked and no `flow_step_runs` row exists with that step id for run `R`
- **THEN** the kernel SHALL throw `ArtifactRegenerationStepUnknown` and SHALL NOT insert any new artifact version

### Requirement: Only affected sections are replaced

For sectioned artifact kinds, `regenerateFromStep` SHALL identify "affected sections" as the set of sections whose `flow_step_id` equals the supplied `step_id`. The kernel SHALL replace the body of every affected section with the output of the replayed step and SHALL NOT invoke any LLM, provider, or syscall on behalf of any non-affected section. For monolithic kinds the entire body SHALL be re-generated by the replayed step (since monolithic kinds have no section-level traceability).

#### Scenario: Affected sections replaced

- **WHEN** `V_old` has sections `[{id:'S1', flow_step_id:'plan'}, {id:'S2', flow_step_id:'search'}, {id:'S3', flow_step_id:'search'}]` and `regenerateFromStep` is invoked for `step_id='search'`
- **THEN** the new version `V_new` SHALL have sections `S2` and `S3` whose body content was produced by the replayed `search` step and `affected_section_ids` SHALL equal `['S2','S3']` (order-insensitive)

#### Scenario: Monolithic kind regenerates entire body

- **WHEN** `regenerateFromStep` is invoked for a `csv_spreadsheet` artifact whose definition is `monolithic`
- **THEN** the kernel SHALL set the entire new-version body to the replayed step output and SHALL leave `affected_section_ids` as the single sentinel value `['__monolithic__']`

### Requirement: Non-affected sections preserved verbatim

For sectioned kinds the kernel SHALL copy every non-affected section from `V_old` into `V_new` byte-for-byte, including `id`, `heading`, `flow_step_id`, `claim_ids`, and body content. The kernel SHALL NOT mutate, re-tokenize, re-summarize, or otherwise transform preserved sections.

#### Scenario: Preserved section bytes identical

- **WHEN** `V_old` section `S1` (not affected by the replayed step) has body `"Intro paragraph"` and metadata `{flow_step_id:'plan', claim_ids:['C1']}`
- **THEN** `V_new` section with `id='S1'` SHALL have body exactly `"Intro paragraph"` and identical metadata

#### Scenario: No syscall fired for preserved sections

- **WHEN** `regenerateFromStep` runs on a version with five sections of which two are affected
- **THEN** the kernel SHALL emit zero LLM or provider syscalls attributable to any of the three non-affected section ids

### Requirement: regeneration_metadata recorded on new version

Every regenerated version SHALL persist a `regeneration_metadata` JSON column on `artifact_versions` with shape `{ replayed_step_id: string, policy_override?: object, provider_override?: object, triggered_at: ISO8601, affected_section_ids: string[] }`. Fresh (non-regenerated) versions SHALL have `regeneration_metadata IS NULL`.

#### Scenario: Metadata captured for regeneration

- **WHEN** `regenerateFromStep({ flow_run_id:'R', step_id:'search', policy_override: { temperature: 0.2 } })` produces `V_new`
- **THEN** `V_new.regeneration_metadata` SHALL deep-equal `{ replayed_step_id: 'search', policy_override: { temperature: 0.2 }, affected_section_ids: [...], triggered_at: <timestamp> }`

#### Scenario: Fresh version has null metadata

- **WHEN** the runtime persists a version produced by a normal (non-regeneration) flow run completion
- **THEN** that row SHALL have `regeneration_metadata IS NULL`

### Requirement: Atomic failure leaves prior version untouched

If any phase of `regenerateFromStep` throws — step replay, section validation, body serialization, R2 upload, D1 insert — the kernel SHALL roll back any partial writes via a single D1 transaction and SHALL NOT mutate the prior version `V_old`, its `status`, its section-status rows, or any R2 object previously associated with `V_old`. The call SHALL re-throw the underlying error so the caller can surface it.

#### Scenario: Step replay failure leaves V_old intact

- **WHEN** `regenerateFromStep` is invoked and the replayed step throws on every retry
- **THEN** the kernel SHALL throw the underlying error, SHALL NOT insert any `artifact_versions` row, and a subsequent `getVersionChain` SHALL return the same chain (ending at `V_old`) as before the call

#### Scenario: R2 upload failure rolls back D1 insert

- **WHEN** `regenerateFromStep` produces a large new body, the prepared D1 insert succeeds in the transaction, and the R2 PUT fails before commit
- **THEN** the transaction SHALL be rolled back, no `artifact_versions` row SHALL be persisted, and the original `V_old.status` and `V_old.body_r2_key` SHALL be unchanged

### Requirement: Policy and provider override scoping

`policy_override` and `provider_override` SHALL apply only to the replay of the named `step_id` and SHALL NOT leak into any other step, sub-run, or future flow execution. The kernel SHALL pass overrides into the `agent-flow` runtime via an isolated override context that is discarded after the replayed step terminates. Other agents or steps invoked indirectly by the replayed step SHALL observe their normal default policy/provider unless they explicitly inherit overrides.

#### Scenario: Override applies to replayed step only

- **WHEN** `regenerateFromStep({ ... step_id: 'search', policy_override: { temperature: 0.1 } })` runs and the replayed `search` step is followed by no further steps in the regeneration scope
- **THEN** the `agent_runs` row for the replayed `search` step SHALL record `policy_json.temperature=0.1` and no other `agent_runs` row in the system SHALL record that override

#### Scenario: Provider override does not leak across runs

- **WHEN** a regeneration with `provider_override: { llm: 'claude-haiku' }` completes
- **THEN** the next normal flow run for the same flow SHALL use the default provider declared on the flow definition, not `claude-haiku`
