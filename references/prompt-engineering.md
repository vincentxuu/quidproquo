# Prompt Engineering 參考

## System Prompt 三段式結構

### 1. 角色定義（Role）
```
你是一個 [具體角色]，專精於 [領域]。你的目標是 [具體目標]。
```
- 具體 > 抽象：「你是資深 RAG 工程師」比「你是 AI 助手」好
- 角色要和任務匹配，不要給無關的身份

### 2. 行為準則（Guidelines）
```
## 回答規則
- 不確定時說「我不確定」，不要猜
- 先給結論，再解釋原因
- 程式碼要可執行，不要虛擬碼
```
- 用列表，不要用長段落
- 正面描述（要做什麼）比負面描述（不要做什麼）有效
- 但關鍵禁止事項要明確列出

### 3. 輸出格式（Format）
```
## 輸出格式
回答請包含：
1. 一句話結論
2. 技術細節（含程式碼）
3. 注意事項
```
- 明確的結構讓輸出一致
- 用範例 + 格式定義雙重約束

## Context 格式化原則

### 語意清晰優於原始拼接
不好：
```
Document 1: ... Document 2: ... Document 3: ...
```

好：
```
## 相關文件

### [文件標題]（來源：xxx，相關度：高）
[內容摘要]

### [文件標題]（來源：yyy，相關度：中）
[內容摘要]
```

### 標記來源和相關度
- 讓 LLM 知道哪些資訊更可信
- 有助於衝突時的判斷

### Token 預算管理
- 預留 30% context window 給生成
- 最相關的文件放前面（primacy effect）
- 超過預算時先摘要再送入

## 信心機制

在 system prompt 中加入：
```
如果提供的資料不足以回答問題，請明確說明「根據目前的資料無法確定」，
並說明還需要什麼資訊。不要猜測或編造答案。
```

## 推理框架

### Chain-of-Thought（CoT）
```
請一步一步思考這個問題：
1. 先分析...
2. 然後考慮...
3. 最後得出結論...
```
適合：數學推理、邏輯問題、多步驟分析

### Few-shot
```
以下是幾個範例：

輸入：X → 輸出：Y
輸入：A → 輸出：B

現在請處理：輸入：C → 輸出：？
```
適合：格式轉換、分類、風格模仿

### ReAct
```
Thought: 我需要先查詢...
Action: search("...")
Observation: 找到了...
Thought: 根據結果，我需要再...
Action: search("...")
Observation: ...
Answer: 綜合以上資訊...
```
適合：需要多步驟檢索的問題

## Prompt 迭代方法

1. **從最簡單的 prompt 開始**
2. **用真實 case 測試**，不要用假設的例子
3. **發現錯誤 → 分析是哪一類**（理解錯誤？格式錯誤？知識不足？）
4. **針對性修改**，不要大改
5. **記錄每次改動和結果**，prompt 也需要版本控制
6. **用 LLM-as-Judge 做快速評估**，但不要完全依賴

## 常見錯誤

- 在 prompt 裡塞太多規則 → LLM 會忽略後面的
- 用否定句（「不要 X」）→ 改用肯定句（「當遇到 X 時，做 Y」）
- 沒有給範例 → 格式不穩定
- prompt 太長 → 重點被稀釋，拆成多輪對話
